///////////////////////////////////////////////////////////////////////////////////////////
/// This is a "disassembly" of sorts for the manifest file that is currently being tested with.
/// There are six cards in this manifest, and each one has its text expanded below.
/// Over the course of making this file, a couple of possible improvements to the file
/// structure became clear to me. For example:
///	- Card records don't really need type signatures, since all cards of the same
///	  type will be together, and the header tells us how many of each type to expect.
///	- For the same reason, cards don't need end of record signatures. Instead it's
///	  probably better to have signature values between card groups.
///	- In the string table dictionary, all of the offsets can be calculated by adding
///	  the length of the preceding strings to the first offset. Thus, most of the
///	  offset fields are unnecessary. The string length field also could only be
///	  16 bits and we would still be fine.
///	- The string table index field in the Record structure is also likely not needed,
///	  since the indices will be sequential (i.e. the first record will use the first two
///	  indices, etc.) This might have to wait until we start to read in the files though.
///
///	Created by Carl Hyde, 2014
////////////////////////////////////////////////////////////////////////////////////////////
